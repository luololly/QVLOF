# Disk-Resident ANN Integration Layer

This directory records how the QSO and RDO layout artifacts from QVLOF are attached to five external disk-resident ANN systems without vendoring their source trees into this repository.

The branch is organized into the following top-level components:

- `data/`
- `index/`
- `qso/`
- `rdo/`
- `requirements/`
- `scripts/`

The `qso/` subtree contains the static-artifact generator, and `scripts/static_disk_sh/` provides per-baseline wrapper entrypoints.

## Why These Systems Are Documented Separately

The five baselines covered here are public systems with their own build processes, file formats, and licenses:

- PageANN
- SPTAG/SPANN
- Starling
- MARGO
- Gorgeous

The cleanest packaging choice here is to:

- keep the QVLOF implementation in one canonical place
- point to the public upstream repositories of the baseline systems
- document the concrete function-level adaptation points

This preserves reproducibility while keeping each baseline tied to its public upstream build system.

## Common Adaptation Template

Across all five systems, the adaptation logic follows a unified template.

1. **Input data and workload evidence**
   - base vectors
   - query set or workload access profile
   - optional workload windows for dynamic workloads

2. **QSO or RDO preprocessing**
   - QSO produces a static query-aware ordering over layout units
   - RDO produces multiple candidate partial layouts plus a switching policy

3. **Produced layout artifact**
   - page grouping
   - posting-page order
   - partition file
   - relaid disk index
   - graph-replicated placement plan

4. **Baseline-specific import or rebuild**
   - each system has its own builder or relayout stage that consumes the artifact

5. **Online search execution**
   - the online search logic usually remains unchanged
   - the physical layout seen by the search path is the component that changes

## Static QSO Adaptation

For static workloads, QSO acts as a physical layout generator. The baseline system continues to use its own search kernels and scheduling policy, but its disk layout is replaced by a query-aware version.

Typical static insertion points are:

- page graph generation in PageANN
- SSD posting construction in SPANN
- `_partition.bin` generation for Starling
- partition plus relayout for MARGO
- graph-replicated placement generation for Gorgeous

In all five systems, the static pipeline has a unified completed shape:

```text
base vectors + workload access profile
  -> QSO ordering or grouping
  -> baseline-specific layout artifact
  -> builder or relayout import
  -> final on-disk artifact
  -> unchanged baseline search runtime
```

## Dynamic RDO Adaptation

For dynamic workloads, RDO sits above the layout import stage. Instead of producing one final layout, it produces:

- multiple candidate partial layouts
- a controller or switching rule
- replay or online evaluation under workload windows

The baseline search path still executes the ANN queries. RDO changes which layout is active, or which partial layout family should be imported by the target baseline.

In all five systems, the dynamic pipeline has a unified completed shape:

```text
workload windows
  -> candidate layout family generation
  -> switch policy
  -> active layout family per window
  -> unchanged baseline runtime
  -> replay-time evaluation
```

The only system-specific differences are:

- what counts as one layout unit,
- which file or directory prefix identifies the active family,
- which builder or relayout stage materializes the family.

## Layout Units By System

### PageANN

Layout unit: page-aligned vector groups and page graph units.

### SPTAG/SPANN

Layout unit: posting pages and SSD posting-list organization under `BuildSSDIndex`.

### Starling

Layout unit: graph partitions represented by `_partition.bin`, followed by relaid disk sectors in `_disk.index`.

### MARGO

Layout unit: partitioned graph layout plus the relayout output generated by `tests/utils/index_relayout.cpp`.

### Gorgeous

Layout unit: graph-split storage, graph-replicated disk layout, and graph-cache-aware sectors controlled by `split_graph` and `gr_layout`.

## Comparison Table

| System | Base layout primitive | QSO injection point | RDO injection point | Does the online search path stay intact? | Main rebuild artifact |
| --- | --- | --- | --- | --- | --- |
| PageANN | page-aligned grouped vectors | before or during page graph generation | before repeated page-layout regeneration over workload windows | mostly yes | page grouping and final page-based disk layout |
| SPTAG/SPANN | SSD posting pages | before `BuildSSDIndex` finalizes posting layout | before window-specific posting layouts are built and selected | yes | SSD posting organization and search config |
| Starling | graph partition pages | before `_partition.bin` is written | above multiple partition files and relayout choices | yes | `_partition.bin` and relaid `_disk.index` |
| MARGO | partitioned graph sectors | before partition file emission or relayout input | above multiple partition families and relayout outputs | yes | partition file and relayout result |
| Gorgeous | graph-prioritized and graph-replicated pages | before `split_graph` or `gr_layout` finalization | above multiple graph-aware partial layouts | yes | split graph artifacts and graph-replicated index |

## Artifact Contracts

The adaptation notes use the following artifact contracts consistently.

- **PageANN:** page grouping or page-aligned vector order
- **SPANN:** posting-page organization and `BuildSSDIndex` inputs
- **Starling:** `_partition.bin` and relaid `_disk.index`
- **MARGO:** partition file and `index_relayout` output
- **Gorgeous:** graph split artifacts, graph-replicated layout artifacts, and search configuration switches

## Per-System Notes

Each subdirectory records:

- the public upstream source URL
- the relevant local workflow entrypoints used to inspect that system
- a QSO mapping
- an RDO mapping
- a function-level discussion of where the layout artifact enters the original implementation

These notes describe how QVLOF layout outputs couple into existing disk-resident ANN pipelines.

The per-system integration notes live under `index/` alongside their baseline-specific artifact contracts.
